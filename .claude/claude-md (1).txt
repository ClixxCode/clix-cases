# CLAUDE.md — Clix Proposal Builder

## Purpose
This agent powers an AI-guided proposal system that scales to hundreds of cases. Each proposal at proposals.clix.co is a compelling, data-driven, brand-matched digital experience — built from structured client context, audit intelligence, and service prescriptions. The agent scaffolds cases, interprets context, runs audits, and synthesizes everything into a polished proposal that is immediately e-signature ready.

Never fabricate data. Never generate placeholder copy in a published proposal. Every claim must trace back to a context file or Supabase field.

---

## Architecture Constraints
- **Supabase (ops schema)** is the operational control plane — case metadata, file completion status, selected services, theme data, and signature state all live here
- **clix-cases (GitHub repo)** is the content source of truth — all .md and .mdx files live here, read via GitHub API
- **clix-proposals (this repo)** is the rendering platform — never modified by case activity
- All Supabase and GitHub API calls are server-side only
- Assets (logos, screenshots) are stored in Supabase Storage and referenced by URL — no binary files in Git
- Do not modify files outside the active case folder in clix-cases
- Do not change status to `published` — human action only

## Skill Routing

Skills are located in `.claude/skills/`. Invoke the correct skill per task.

| Task | Skill File |
|---|---|
| Create new case folder + manifest | `case-scaffolder.md` |
| Parse uploaded files → extract goals, pain points, talking points | `context-interpreter.md` |
| Map interpreted context → recommended services | `service-prescriber.md` |
| Scrape client site for brand/tone | `brand-extractor.md` |
| Crawl domain → tech stack, issues, opportunities | `domain-enrichment.md` |
| Run SEO audit (SEMrush MCP) | `seo-audit.md` |
| Run technical audit (Firecrawl + Lighthouse) | `technical-audit.md` |
| Run competitor analysis | `competitor-audit.md` |
| Validate manifest + Supabase case_files before build | `manifest-validator.md` |
| Build final proposal.mdx | `proposal-builder.md` |

## Case Intake Flow

### Stage 1 — Case Creation (Clix Pulse)
Required Supabase fields before scaffolding begins:
- `business_name`, `industry`, `primary_domain`
- `context_files[]` — uploaded PDFs, TXTs, DOCXs, etc.

Claude runs `context-interpreter` on all uploaded files and writes back to Supabase:
- `pain_points` (jsonb array)
- `goals` (jsonb array)
- `talking_points` (jsonb array)
- `brief_summary` (text)

Claude then runs `service-prescriber` and writes:
- `ai_recommended_services` (jsonb array — references ops.services schema)

BD reviews and confirms/overrides via Clix Pulse UI → final selection written to:
- `selected_services` (jsonb array)

Selected services influence which audits are queued in `case_files`.

### Stage 2 — Scaffolding
`case-scaffolder` creates the case folder in clix-cases with:
- `manifest.json` (seeded from Supabase case record)
- `brief.md` (generated from Supabase interpreted fields)
- Empty audit placeholders per selected services

### Stage 3 — Audits
Each audit skill writes its output to `cases/{case-id}/audits/{audit-type}.md` and updates `ops.case_files` (completed: true).

### Stage 4 — Domain Enrichment (Manual Trigger)
BD clicks "Enrich Profile" in Clix Pulse → `domain-enrichment` skill runs → returns tech stack, issues, and opportunities for human review → on approval, commits to `cases/{case-id}/enrichment/domain-crawl.md`.

### Stage 5 — Build
`manifest-validator` confirms all expected files in `ops.case_files` are complete → sets `build_ready: true` in Supabase.

`proposal-builder` reads all context files via GitHub API and synthesizes `proposal.mdx`. On completion:
- Updates `ops.cases` (status: draft, last_built_at: now())
- Proposal is e-signature ready immediately

### Stage 6 — Publish
Human reviews draft at proposals.clix.co/case-id (gated). Approves in Clix Pulse → status: published → Supabase webhook hits `/api/revalidate` → ISR revalidates the route → proposal is live.

## Context File Priority (Proposal Build)

Read in this order. Each file adds specificity. Absent files are skipped silently.

1. `brief.md` — required. Goals, pain points, talking points, deal context.
2. `brand-extract.md` — tone, visual direction, messaging pillars.
3. `enrichment/domain-crawl.md` — tech stack, site issues, quick wins.
4. `audits/seo-audit.md` — organic performance, keyword gaps, opportunity sizing.
5. `audits/technical-audit.md` — Core Web Vitals, infrastructure, health score.
6. `audits/competitor-audit.md` — competitive landscape, differentiation angles.
7. `.claude/context.md` — agent decisions, overrides, sensitivities. Never surfaces in proposal output.

Tier is determined automatically:

| Tier | Files Present |
|---|---|
| basic | brief.md only |
| standard | + brand + enrichment or 1 audit |
| premium | + 2 or more audits |

## Proposal Build Rules

- Write in second person — client-facing, confident, specific. Zero filler.
- Every recommendation must cite a specific finding from the context files
- Tone calibrated to industry and brand-extract.md — a law firm and a boat lift manufacturer are not the same voice
- Mockup sections should reflect the client's actual industry and brand colors from theme.json
- Pricing references ops.services schema values — never hardcode

### proposal.mdx Frontmatter
```mdx
---
caseId: case-001
clientName: Multisorb Filtration Group
industry: Industrial Filtration
primaryDomain: multisorb.com
generatedAt: 2026-03-04
tier: premium
status: draft
selectedServices: ["PPC", "SEO", "Web Rebuild"]
signatureStatus: unsigned
signingUrl: /case-001/sign
---
```

Status values: `draft` → `published` → `signed`
Only humans change `published`. Signature API changes `signed`.

## E-Signature Block

Every proposal.mdx includes an ESignature section at the close. The component:
- Displays selected services + total investment summary
- Captures: signer name, title, e-signature, timestamp, IP
- On submit: POSTs to `/api/sign` → writes to `ops.case_signatures` in Supabase
- Updates `ops.cases` (signature_status: signed, signed_at, signed_by)
- No third-party DocuSign dependency for MVP — native implementation

Supabase fields required:
```json
case_signatures {
  case_id, signer_name, signer_title,
  signature_data, signed_at, ip_address, accepted_terms
}
```

## Supabase Schema Reference (ops schema)

**cases**: id, business_name, industry, primary_domain, status, tier, build_ready, last_built_at, pain_points (jsonb), goals (jsonb), talking_points (jsonb), brief_summary, ai_recommended_services (jsonb), selected_services (jsonb), signature_status, created_at

**case_files**: id, case_id, file_type, github_path, completed, generated_at

**case_theme**: case_id, primary_color, secondary_color, font_heading, font_body, logo_url, raw_theme_json (jsonb)

**case_signatures**: id, case_id, signer_name, signer_title, signature_data, signed_at, ip_address, accepted_terms

## Do Not
- Do not publish proposals — human action only
- Do not run audits against a URL not present in brief.md or manifest.json
- Do not infer services not confirmed in `selected_services`
- Do not generate placeholder or Lorem Ipsum content
- Do not call external APIs not listed in the active skill file
- Do not write binary files to clix-cases — use Supabase Storage URLs
- Do not expose `.claude/context.md` content in proposal output
